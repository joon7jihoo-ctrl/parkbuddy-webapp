# ParkBuddy Web 평가분석보고서

작성일: 2026-06-07

## 1. 평가 범위

이번 평가는 한글 중심 UI 개편, 모바일 사용성 개선, 라운딩 생성/조편성/점수 입력 흐름, Supabase 정규 테이블 저장, 실시간 순위 확인, 공유 기능, 개인점수관리 대시보드, 취약점 및 운영 리스크를 대상으로 수행했습니다.

평가 대상:

- 홈, 회원 등록 및 관리, 라운딩 생성, 참가자 선택, 조편성 결과, 라운딩 기록 보기
- 코스별 기준파 입력과 36타 검증
- 공유 링크 기반 코스별 점수 입력 및 순위 동기화
- 신페리오/스트로크/매치/스크램블/포섬/포볼/스테이블포드 순위 계산
- 공유 카드 텍스트/PNG 저장
- 개인점수관리 성과 대시보드
- Supabase 정규 테이블 및 fallback 저장 구조
- 모바일/다크모드 UI 품질

## 2. 구현 결과

UI/UX 반영:

- 화면 노출 문구를 한글 중심으로 정리했습니다.
- 홈 화면의 `PARKBUDDY PERFORMANCE CLUB` 문구는 동호회명으로 대체했습니다.
- 홈 하단 중복 메뉴는 제거하고, 등록 회원/라운딩 기록 카드를 메뉴 진입점으로 유지했습니다.
- 최근 하이라이트 아래에 라운딩 시작/성장 데이터 버튼을 배치했습니다.
- 모바일에서 홈 화면과 라운딩 생성 화면의 스크롤 부담을 줄이도록 단계형 입력 흐름을 적용했습니다.
- 다크모드에서 숫자, 옵션 제목, 입력 카드 문자가 묻히는 문제를 보정했습니다.

라운딩/조편성:

- 기본 홀 수는 9홀부터 81홀까지 9홀 단위로 선택할 수 있습니다.
- 기준파 입력은 A코스, B코스 등 코스별 1~9홀 카드로 입력합니다.
- 각 코스 기준파 합계가 36타가 아니면 해당 코스를 알려주는 팝업을 띄웁니다.
- 모바일 숫자 키패드 입력 시 기준파가 12로 고정되던 문제를 raw 입력 방식으로 수정했습니다.
- 조편성은 참가자 선택, 편성 방식 선택, 조당 인원 선택 단계로 분리했습니다.
- 조당 인원은 3명 이상만 가능하도록 했습니다.
- 사용자 지정 편성을 추가했고, 회원별 조 번호 입력값이 결과 조 번호로 유지됩니다.
- 조장은 후보 여부와 관계없이 선택할 수 있고, 직접 선택한 사람만 편성표에서 조장으로 표시됩니다.
- `조편성하기` 버튼은 화면 하단에 고정되어 긴 목록에서도 바로 실행할 수 있습니다.

점수/순위/공유:

- 라운딩 생성은 조편성 기록 생성까지만 수행하고, 점수 입력은 라운딩 기록 보기에서 시작합니다.
- 점수 입력 화면은 코스별 입력 카드로 구성했습니다.
- 공유 링크 점수 입력 화면은 상단 코스 탭과 1~9홀 표 입력 UI로 구성했습니다.
- 점수 저장 시 라운딩 기록의 점수와 순위가 갱신됩니다.
- 라운딩 기록/공유 점수 화면은 Supabase 저장소를 5초 간격으로 동기화해 실시간 순위를 확인할 수 있습니다.
- 라운딩 결과 공유 텍스트와 PNG 공유 이미지 저장 기능을 추가했습니다.

개인점수관리:

- 회원별 기준파 대비 기록 추이 그래프를 제공합니다.
- 최근 5라운드 평균, 최근 성장 폭, 베스트 배지, 코스별 강약 분석을 추가했습니다.
- 팀전 기록도 구성원별 개인 기록에 반영됩니다.

Supabase:

- 정규 테이블 `parkbuddy_clubs`, `parkbuddy_members`, `parkbuddy_courses`, `parkbuddy_rounds`, `parkbuddy_round_scores`, `parkbuddy_round_rankings`, `parkbuddy_app_settings`를 우선 사용합니다.
- 기존 `parkbuddy_app_state`는 fallback/migration 저장소로 유지합니다.
- 원격 REST 확인 결과 fallback 테이블과 정규 테이블이 모두 200으로 응답했습니다.

## 4. Rounds와 Active Players 판단

`Rounds`는 저장된 라운딩 기록 수이고, `Active Players`는 라운딩 기록에 참여한 고유 회원 수입니다. 의미는 다르지만 현재 홈 화면에서는 등록 회원 수와 라운딩 기록 수가 운영자가 바로 행동할 수 있는 핵심 지표입니다. 혼선을 줄이기 위해 `Active Players` 카드는 제거하고 `등록 회원`, `라운딩 기록` 2개 카드만 남겼습니다.

## 5. 실시간 순위 확인 방법

운영자는 라운딩 기록 보기에서 해당 라운딩의 `점수 입력` 또는 `점수 수정`을 눌러 직접 점수를 입력할 수 있습니다. 회원에게는 `조편성/점수 링크 공유` 버튼으로 링크를 전달합니다. 회원이 공유 링크에서 본인 또는 팀을 선택해 점수를 저장하면 라운딩 기록의 순위가 갱신되고, 기록 화면은 5초 간격으로 Supabase 데이터를 다시 불러와 경기 방식별 순위를 표시합니다.

현재 구조는 주기적 동기화 기반입니다. 수백 명 동시 접속 수준으로 확장하려면 Supabase Realtime 채널을 연결해 `parkbuddy_round_scores` 변경 이벤트를 구독하는 방식으로 바꾸는 것이 좋습니다.

## 6. 취약점 및 UI/UX 정밀 분석

### 디자인/UX 영역

High Quick Win:

- 문제: 전체 그리드 스코어 입력은 기능적으로 명확하지만, 작은 화면에서 숫자 셀이 많아지면 업무용 표처럼 보일 수 있습니다.
- 해결: 현재처럼 코스 탭을 유지하되 셀 안에는 홀 번호, 파, 입력값만 남기고 색상은 언더/이븐/오버 상태에만 제한합니다. 입력 완료율, 현재 순위, 기준파 대비 값은 상단 고정 요약으로 유지합니다.
- 권장 가이드: Pretendard 또는 system-ui 기반 굵은 숫자, 초록/라임 포인트, 경고는 오렌지/레드만 사용, 카드 반경 8px 이하 유지.

High Quick Win:

- 문제: 고령층과 젊은 층을 모두 만족시키려면 버튼은 크고 명확해야 하지만, 젊은 층에게는 지나치게 행정 도구처럼 보이면 매력이 떨어집니다.
- 해결: 운영 입력 화면은 큰 터치 영역을 유지하고, 결과/공유/개인 데이터 화면은 스포츠 앱처럼 숫자 중심, 랭킹 중심, 배지 중심으로 구성합니다.
- 적용: 홈 대시보드, 공유 카드, 개인점수관리 배지를 추가했습니다.

Medium Quick Win:

- 문제: Shareable Card가 텍스트 공유에만 머물면 인스타그램 공유 매력이 낮습니다.
- 해결: PNG 저장 기능을 추가했고, 카드에는 라운딩명, 장소, 날짜, 우승자, 기준파 대비 성적, 상위 3명, 앱 슬로건을 포함했습니다.
- 장기 개선: `html-to-image` 또는 `satori`를 도입해 DOM 기반 카드 템플릿을 만들면 디자인 품질과 유지보수가 좋아집니다.

Medium 장기 과제:

- 문제: 현재 AI 조편성 UX는 편성 이유를 보여주지만 사용자가 조정 가능한 "추천 근거"가 더 필요합니다.
- 해결: 조편성 결과에 "왜 이렇게 편성됐는지"를 실력 편차, 직전 중복, 선택 조장 분포, 3인 조 보정으로 나누어 보여줍니다.
- 권장 문구: "14명은 4·4·3·3 구조가 가장 안정적이에요", "중복 조합 2건을 피해서 편성했어요".

Low 장기 과제:

- 문제: 다크모드는 현재 색상 보정 중심입니다.
- 해결: 야간 라운딩 전용 모드로 배터리 절약, 큰 숫자, 고대비 초록/라임 포인트를 별도 테마 토큰으로 분리합니다.

### 기술/운영 영역

High Quick Win:

- 문제: 여러 명이 같은 조 점수를 동시에 수정하면 마지막 저장이 이전 저장을 덮어쓸 수 있습니다.
- 해결: 현재는 정규 테이블과 5초 동기화로 확인 가능하지만, 충돌 방지를 위해 `updated_at` 기반 optimistic locking 또는 entry 단위 upsert를 강화해야 합니다.
- 권장 구현: `parkbuddy_round_scores`에 `updated_by`, `updated_at`, `version`을 두고 저장 시 버전이 다르면 재확인 안내를 띄웁니다.

High Quick Win:

- 문제: 정규 테이블 전환 중 기존 JSON 상태와 정규 테이블 데이터가 동시에 존재하면 마이그레이션 기준이 꼬일 수 있습니다.
- 해결: `parkbuddy_app_settings`에 `schema_mode`, `migrated_at`, `state_key`를 기록하고, 정규 테이블 데이터가 있으면 정규 테이블을 우선 읽도록 유지합니다.
- 현재 상태: 정규 테이블 우선, legacy fallback 구조가 적용되어 있습니다.

High 장기 과제:

- 문제: 현재 RLS 정책은 MVP 운영 편의를 위해 anon 접근이 넓습니다.
- 해결: Supabase Auth 도입 후 `club_id`와 사용자 역할을 연결하고 관리자만 전체 동호회, 일반 사용자는 가입 동호회만 조회하도록 RLS를 좁혀야 합니다.
- 권장 정책: `club_memberships(user_id, club_id, role)` 테이블을 기준으로 `select/insert/update/delete` 정책을 분리합니다.

Medium Quick Win:

- 문제: 모바일 네트워크가 불안정할 때 점수 저장 실패가 사용자에게 늦게 드러날 수 있습니다.
- 해결: 저장 상태 배지, 재시도 큐, 마지막 동기화 시간을 표시합니다.
- 권장 라이브러리: TanStack Query의 mutation retry, 또는 IndexedDB 기반 `idb-keyval` 임시 큐.

Medium 장기 과제:

- 문제: 노쇼, 중도 이탈, 코스 변경 같은 현장 변수는 현재 기록 수정으로 우회해야 합니다.
- 해결: 라운딩 기록 상세에서 참가자 제외, 조 이동, 코스 재배정, 점수 잠금 해제 플로우를 추가합니다.

Medium 장기 과제:

- 문제: 동시 접속자가 수백 명으로 늘면 GitHub Pages 정적 배포는 문제 없지만 Supabase REST 저장과 5초 polling이 병목이 될 수 있습니다.
- 해결: Supabase Realtime 구독, score row 단위 저장, 라운드별 채널 분리, Edge Function 집계 캐시를 도입합니다.

Low 장기 과제:

- 문제: React Router가 아직 도입되지 않아 화면별 URL 복원과 브라우저 새로고침 UX가 제한적입니다.
- 해결: `react-router-dom`을 도입해 `/records/:recordId`, `/score/:recordId/:entryId` 경로를 구성합니다.

## 7. 우선순위별 개선안

High:

- Supabase Auth와 `club_memberships` 기반 RLS 적용
- 점수 저장 충돌 방지를 위한 row-level version 관리
- 모바일 저장 실패 재시도/오프라인 큐
- 라운딩 기록에서 참가자/조/코스 변경 기능

Medium:

- Supabase Realtime으로 순위표 push 갱신
- 공유 카드 DOM 템플릿화 및 인스타그램 최적 비율 템플릿 추가
- 카카오 JavaScript SDK 공유, 문자 발송은 Edge Function + 메시지 사업자 API로 확장
- React Router 도입
- 개인 대시보드에 코스별 버디율, 파 세이브율, 최근 5회 추세 라벨 추가

Low:

- 다크모드 전용 팔레트 세분화
- 라운딩 종료 후 MVP/오늘의 베스트 샷 같은 감성 배지
- 관리자용 운영 리포트 PDF 출력
- 경기장/코스별 평균 난이도 통계

## 8. 결론

이번 수정으로 ParkBuddy Web은 단순 운영 도구에서 파크골프 데이터 스포츠 플랫폼에 가까운 구조로 개선되었습니다. 한글 중심 UI, 단계형 모바일 흐름, 사용자 지정 편성, 선택 조장 표시, 코스별 기준파 검증, 실시간 점수 동기화, 공유 이미지 저장, 개인 성과 대시보드가 반영되었습니다.

남은 핵심 과제는 인증/권한/RLS, 실시간 충돌 방지, 오프라인 저장 방어, 현장 변수 대응 기능입니다. 이 네 가지가 보강되면 동호회 단위 MVP를 넘어 여러 동호회를 관리하는 운영 플랫폼으로 확장할 수 있습니다.
